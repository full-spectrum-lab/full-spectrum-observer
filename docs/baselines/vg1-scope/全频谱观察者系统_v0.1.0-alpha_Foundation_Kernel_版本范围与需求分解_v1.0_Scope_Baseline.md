## 1. 版本目标与成功定义

### 1.1 主目标

本版本只证明 Foundation Kernel 成立，不证明普通用户可用、企业可用或治理控制面完整。

### 1.2 成功定义

版本通过时必须同时满足：

1. 在断网、无模型、无遥测、无企业 Connector 的环境中运行；
2. 至少一种开发者输入路径可发起分析；
3. Adapter、Schema Validation、Governance Validation 职责分离；
4. 每次运行形成不可变 RuntimeConfigurationSnapshot；
5. Engine 通过 Facade 被调用，System 不复制正式治理算法；
6. Output 保留 UNKNOWN、Evidence、Profile/Policy/Knowledge/Subject/Version 引用；
7. 高总分不得覆盖 PROHIBITED、UNKNOWN、Human Anchor 或硬 Gate；
8. 产生 ObservationRecord 和追加式 AuditEvent；
9. Snapshot/Audit 篡改可被检测；
10. 第二干净环境走查通过。

### 1.3 不代表

本版本不代表：

- 有完整 Console；
- 普通用户可以独立使用；
- 已有 Copilot；
- 已有主体、知识、Profile、I/O 可视化管理；
- 已有企业审批、R4 或 Connector；
- 已有 Exact Replay；
- 已完成外部验证、企业 PoC、生产验证或合规认证。

| 约束 ID | 名称 | 规范强度 | 约束 | 禁止 | 版本验收 |

## 8. 端到端流程与状态

| 流程 | 名称 | 序列 | 预期结果 |
| --- | --- | --- | --- |
| FLOW-01 | 成功分析 | Request → Intake → Adapter → Schema Validation → Governance Validation → Snapshot → Engine Facade → Engine → Output Assembler → Observation/Audit | COMPLETED + output_ref + audit_head |
| FLOW-02 | Schema 失败 | Request → Intake → Adapter/Raw Schema Validation → FAILED_SCHEMA | Engine 不调用；field_path + reason_code + Audit |
| FLOW-03 | Governance 不充分 | Schema PASS → Governance Validation → UNKNOWN/NEEDS_EVIDENCE 或按 Profile 阻断 | 不得转成 0/低风险；输出或失败状态有 Audit |
| FLOW-04 | Adapter 失败 | Request → Adapter Exception/Mapping Error | 不生成成功 Context；保留 input digest、错误字段、reason_code、Audit |
| FLOW-05 | 幂等重试 | 相同 idempotency_key + 等价请求 | 返回既有 operation/observation，不重复成功记录 |
| FLOW-06 | 完整性验证 | 读取 Snapshot/Audit → canonicalize → recompute digest/hash | 篡改可检测；返回验证结果与 reason_code |
| FLOW-07 | 离线执行 | 断开外网并禁用所有外部 Provider/Connector | 完整 Foundation CASE 可运行，且无外部连接尝试 |


### 8.1 最小状态机

```text
RECEIVED
→ INTAKE_VALIDATED
→ ADAPTING
→ SCHEMA_VALIDATING
→ GOVERNANCE_VALIDATING
→ SNAPSHOT_FIXED
→ ENGINE_RUNNING
→ OUTPUT_ASSEMBLING
→ COMPLETED

任一步可进入：FAILED / TIMED_OUT / CANCELLED（若详细设计支持取消）
```

### 8.2 幂等规则

同一 `idempotency_key` 和等价请求不得产生多个成功 Observation。非等价请求复用同一键必须明确拒绝。

## 10. 数据所有权与持久化

| 数据对象 | 所有者 | 持久化 | v0.1 规则 | 安全边界 |
| --- | --- | --- | --- | --- |
| Raw Input | Input Intake / caller | 默认不持久化 | 仅记录 digest、kind、size、source；synthetic Fixture 可保留完整内容 | 不得进入日志/错误详情 |
| GovernanceEvent / CanonicalContext | Execution Pipeline | 最小化；默认仅保留 digest 或脱敏制品 | 详细设计必须决定是否保存脱敏 Context | 不声明 Exact Replay |
| RuntimeConfigurationSnapshot | Execution / Evidence | 必须持久化 | 不可变；固定所有版本与 digest | 完整性校验必须可执行 |
| GovernanceOutputEnvelope | Observation Store / Artifact Store | 必须持久化 | 保存原始正式输出；报告/展示不得覆盖 | 敏感扩展需分类 |
| ObservationRecord | Observation Store | 必须持久化 | 保存状态、引用、幂等和 trace | 可查询但不是 Audit 账本 |
| AuditEvent | Audit Ledger | 追加式持久化 | canonical hash + previous_hash；禁止更新/删除 | 允许隐私 payload 外置，仅保留摘要 |
| Logs | Operations | 短期本地保留 | 结构化、可关联 trace_id | 不得含秘密或完整 Raw Input |
| Secrets | Secret Handling | 默认无秘密；需要时只存安全引用 | 配置导出不得包含明文 | 不得进入日志/报告 |


### 10.1 Replay 声明

v0.1 不承诺 Exact Replay。只有 Snapshot、必要输入/Context 制品、Engine/Schema/Pack 依赖和执行环境全部满足后，后续版本才能声明相应 Replay 等级。

## 11. Audit 与完整性

### 11.1 最低 AuditEvent

至少包含：

```text
event_id
event_type
event_time
observation_id / operation_id / trace_id
actor/service/subject refs（适用时）
stage
reason_codes
payload_digest
previous_hash
event_hash
serialization_version
```

### 11.2 完整性规则

- 使用固定版本 canonical serialization；
- event_hash 至少采用 SHA-256 或等价密码学摘要；
- previous_hash 构成可验证链；
- Ledger 为追加式；
- 验证命令能够检测 Snapshot 或 AuditEvent 被修改；
- 并发写入和 hash chain 分区策略由 VG2 决定。

## 12. Alpha 安全与隐私边界

本版本最低要求：

1. 默认关闭公网和远程模型；
2. 默认关闭遥测、企业写回和 R4；
3. 无外部连接目标；
4. Raw Input 默认不长期保存；
5. 日志、错误、报告不包含秘密或完整敏感输入；
6. 默认运行尽量不需要秘密；
7. 业务数据只作为数据，不作为系统指令；
8. Engine 不持有企业凭证；
9. synthetic Fixture 与真实业务证据分级标记；
10. 依赖和发布包可生成固定版本清单。

### 13.1 AT-09 硬门控验收

必须存在一个 Golden CASE：即使综合分数较高，只要出现 PROHIBITED、UNKNOWN 或必须 Human Anchor 的硬条件，Output 仍保留该条件并进入相应状态。

### 13.2 第二干净环境

**非主要开发者**指未参与 `v0.1.0-alpha` 功能实现、安装打包或测试脚本编写的人。可以是同一团队成员或外部参与者，但不得是主要开发者本人，也不得由主要开发者实时逐步指导。

该人员必须在与主要开发环境隔离的独立机器或全新虚拟环境中，仅依赖版本随附文档完成：

```text
解压/安装
→ 配置
→ 运行 Reference CASE
→ 查看 Output 与 Audit
→ 验证离线
→ 验证完整性
→ 停止与卸载/清理
```

走查不得使用主要开发机已有缓存、私有配置或未发布脚本，必须记录环境、步骤、失败、修正和最终结果。该证据属于内部独立环境证据，不等于 v0.3 非作者用户验证或 v0.6 企业 PoC。
